**この2つが混同されるだけで、チームの品質活動はズレていく**

「テスト計画を作って」と言われて、あなたが作るものは何ですか？

テストのスケジュール？ テストケースの一覧？ 担当者の割り当て？ それとも、リスク分析やテスト方針？

もし「全部ごちゃまぜにしたドキュメント」を思い浮かべたなら、あなたは多数派です。そして、**その混同こそが、テストが迷走する根本原因**です。

この記事では、「テスト戦略」と「テスト計画」の違いを、できるだけシンプルに整理します。専門用語を振りかざすつもりはありません。現場で「あ、そういうことか」とスッキリしてもらえることがゴールです。

:::message
**この記事を読んでほしい人**
- テスト戦略とテスト計画の違いがうまく説明できない人
- テスト計画書に何を書けばいいか毎回迷う人
- PM・開発者・QA——テストに関わるすべてのロールの人
:::

## 30秒でわかる違い

まず結論から。

| | テスト戦略 | テスト計画 |
|---|---|---|
| 問い | **「何を、なぜテストするか」** | **「いつ、誰が、どうやるか」** |
| 内容 | 守るべき品質を決める | スケジュールと担当 |
| | リスクに基づく優先順位 | テスト環境・ツール |
| | テストの方針・考え方 | 具体的な作業手順 |
| | プロジェクト全体に適用 | 個々のテスト活動に適用 |

一言で言えば、**戦略は「方針」、計画は「段取り」**です。

## たとえ話で理解する

旅行にたとえるとわかりやすくなります。

![旅行のたとえ：戦略と計画](/images/test-plan-vs-strategy/travel-analogy.png)

| | テスト戦略 | テスト計画 |
|---|---|---|
| 旅行で言うと | 旅の目的地と目的 | 旅の段取り |
| 決めること | どこへ、なぜ行くか | どうやって、いつ行くか |
| 例 | 「京都に行こう。目的は寺社仏閣と食文化の体験」 | 「朝9時の新幹線、初日は清水寺、宿は四条」 |

「京都に行く」が決まっていれば、新幹線の時間もホテルも自然に決まります。でも、目的地が「京都」なのか「沖縄」なのかが曖昧なまま段取りだけ進めたら、チームはバラバラに動きます。

> 計画は変わる。でも戦略がブレなければ、
> 計画が変わっても迷子にはならない。

## もう少し正確に——比較表

実務で使えるレベルまで、両者の違いを掘り下げます。

| 比較軸 | テスト戦略 | テスト計画 |
|---|---|---|
| **答える問い** | Why / What — なぜ、何をテストするか | How / When / Who — どう、いつ、誰がテストするか |
| **スコープ** | プロジェクト全体 or プロダクト全体 | 個別のリリース、スプリント、テストフェーズ |
| **寿命** | 長い（プロジェクトを通じて有効） | 短い（リリースやスプリントごとに更新） |
| **変更頻度** | 低い（大きな方針転換時のみ） | 高い（毎スプリント or 毎リリース） |
| **主な作成者** | QAリード、テストマネージャー、PM | テスター、QAエンジニア |
| **合意すべき相手** | PM、開発リード、ステークホルダー | テストチーム、開発チーム |
| **含まれる内容** | 品質目標、リスク分析、テストレベル方針、終了条件 | スケジュール、担当割り当て、テスト環境、テストケース一覧 |

:::message
**ポイント：寿命の違い**
テスト戦略はプロジェクトを通じて生き続けます。テスト計画はスプリントやリリースごとに更新されます。この「寿命の違い」を意識するだけで、何をどちらに書くべきか迷わなくなります。
:::

## 混同するとどうなるか

「別に分けなくても、1つのドキュメントにまとめれば良くない？」——こう思うかもしれません。実際、多くの現場では1つの「テスト計画書」にすべてが詰め込まれています。

でも、混同にはコストがあります。

### 症状1：毎回ゼロからテスト計画を作っている

:::details テスト戦略がない現場
**リリースA：** テスト計画を作る。リスク分析もテスト方針もスケジュールも全部書く。30ページ。

**リリースB：** また1からテスト計画を作る。リスク分析もテスト方針もスケジュールも全部書く。25ページ。

**リリースC：** 前回のをコピペして日付だけ変える。リスク分析は前回のまま。形骸化。
:::

テスト戦略を分離していれば、方針部分は使い回せます。毎回のテスト計画は「今回の段取り」だけに集中でき、**5ページで済む**ようになります。

### 症状2：テスト方針がリリースごとにブレる

:::details テスト方針が計画に埋もれている場合
**リリースA：** 「決済の信頼性を最優先」と計画書に書く

**リリースB：** PMが変わり「UIの使いやすさを重点テスト」に。Aの方針は参照されない

**結果：** プロダクトとして一貫した品質方針がなく、リリースごとにテストの重心が漂流する
:::

テスト戦略が独立したドキュメントとして存在していれば、PMが変わっても、チームメンバーが入れ替わっても、**品質の方針は引き継がれます**。

### 症状3：誰が何を決めるのかわからない

1つのドキュメントにすべてが入っていると、「この部分は誰が決めるの？」が曖昧になります。

| ロール | 責任領域 | 説明 |
|---|---|---|
| **PM** | 戦略 | 「何を守るか」はビジネス判断。PMが関与すべきは**戦略**の領域。 |
| **QA** | 計画 | 「どうテストするか」は専門判断。QAが主導すべきは**計画**の領域。 |
| **開発** | 戦略・計画の両方 | テストレベルの分担、自動テストの範囲。**戦略・計画の両方**に関わる。 |

戦略と計画を分けることで、**「この部分はPMと合意が必要」「この部分はテストチームで決められる」**という判断が明快になります。

## それぞれに何を書くか

具体的に、テスト戦略とテスト計画にはどんな項目が入るのかを整理します。

### テスト戦略に書くこと

1. **品質目標** — このプロダクトで守るべき品質は何か
   例：「決済処理の信頼性」「個人情報のセキュリティ」「初回ユーザーの操作完了率」

2. **リスク分析と優先順位** — どこにリスクが高く、どこに集中するか
   例：「外部API連携は障害リスクが高いため、重点テスト領域とする」

3. **テストレベルの方針** — どのレベルで何を担保するか
   例：「ビジネスロジックは単体テストで網羅。UIフローはE2Eで主要パスのみ」

4. **テスト観点の方針** — 重視する観点と軽量化する観点
   例：「異常系・境界値を重視。正常系の単純確認は自動化で対応」

5. **終了条件** — テストの「完了」をどう定義するか
   例：「重大バグ0件、主要シナリオ全通過、ステークホルダーの合意」

### テスト計画に書くこと

1. **スケジュール** — いつ、何をやるか
   例：「第2週：結合テスト、第3週：システムテスト、第4週：受け入れテスト」

2. **担当者** — 誰が何を担当するか
   例：「決済テスト：Aさん、認証テスト：Bさん、回帰テスト：自動化（CI）」

3. **テスト環境** — どの環境でテストするか
   例：「ステージング環境（本番相当）、テストデータはXXを使用」

4. **テストケース** — 具体的に何を確認するか
   例：「テストケース一覧（Qase / スプレッドシートへのリンク）」

5. **リスクと対策** — このリリース固有のリスクへの対応
   例：「テスト環境の不安定さ → 代替環境を確保済み」

:::message
**戦略は「1枚」、計画は「毎回」**
テスト戦略は理想的には1ページに要約できます。テスト計画はリリースやスプリントごとに更新します。この運用イメージを持つだけで、両者の関係がクリアになります。
:::

## 2つの関係——戦略が計画を導く

テスト戦略とテスト計画は、独立した別物ではありません。**戦略が計画の上位にあり、計画は戦略を具体化したもの**です。

![戦略と計画の階層関係](/images/test-plan-vs-strategy/strategy-plan-hierarchy.png)

> **戦略（方針）**
> 「決済の信頼性を最優先で担保する。異常系テストを厚くし、決済フローのE2Eテストを自動化する」

> **計画（段取り）**
> 「v2.3リリースでは、決済API変更部分の異常系テスト20件を第2週に実施。担当はAさん。E2Eテスト5シナリオをCIに追加」

この関係が理解できれば、テスト計画を作るときに毎回「何を書けばいいんだっけ」と迷うことがなくなります。**戦略で決まっている方針を、今回のリリースに当てはめるだけ**だからです。

## 「うちにはテスト戦略がない」——どうすればいい？

多くの現場では、テスト戦略が明文化されていません。テスト計画書だけが毎回作られ、方針はチームリーダーの頭の中にだけある。これは珍しくありません。

その場合、**いきなり立派な戦略書を作ろうとしないでください**。以下の5つの問いに答えるだけで、最小限のテスト戦略ができあがります。

- このプロダクトで「壊れたら最も困る」機能はどれか？
- 過去のリリースで、最も痛かったバグはどこで起きたか？
- テストで重点的に見るべき領域はどこか？ 逆に軽くしていい領域は？
- どのテストレベル（単体 / 結合 / システム / 受け入れ）で、何を担保するか？
- テストが「完了した」と言える条件は何か？

これを**A4一枚**にまとめて、PMと開発リードに共有する。それだけで、テスト計画を作るときの判断基準ができます。

:::message
**完璧を目指さない**
最初のテスト戦略は50点でいい。大事なのは「方針が明文化されていること」そのものです。文字になっていれば、チームで議論でき、改善できます。頭の中にあるだけでは、議論すらできません。
:::

## 各ロールにとっての意味

テスト戦略と計画の区別は、ロールによって「効いてくる場面」が違います。

| ロール | 戦略がないと困ること | 計画がないと困ること |
|---|---|---|
| **PM** | リリース判断の根拠がない。「大丈夫？」と聞くしかなくなる | 進捗が見えない。テストがいつ終わるかわからない |
| **開発者** | 単体テストでどこまでカバーすべきかわからない | いつテスト環境を渡せばいいかわからない |
| **QA** | テストの優先順位が決められない。全部やる羽目になる | 日々の作業が場当たり的になる |
| **EM** | テストチームの人員配置・スキル育成の方向が定まらない | チームのキャパシティと作業量のバランスが取れない |

## よくある質問

### Q. 小さいプロジェクトでも分ける必要ある？

あります。ただし、**ドキュメントの量は変わります**。小さいプロジェクトなら、テスト戦略は箇条書き5行で十分。テスト計画もチケット1枚分で事足ります。分離することの意味は、ドキュメントの量ではなく、**「考える順番」を正しく保つこと**にあります。

### Q. アジャイルでも戦略は必要？

むしろアジャイルのほうが必要です。スプリントごとにテスト計画は変わりますが、**品質の方針はスプリントをまたいで一貫している**べきです。戦略がなければ、スプリントごとにテストの重心が漂流し、チーム全体の品質感覚がバラバラになります。ただし、アジャイルには固有の運用上の注意点があります。詳しくは次のセクション「アジャイルでの落とし穴——マンネリ化を防ぐ」で解説します。

### Q. テスト戦略は誰が書くの？

理想的にはQAリードやテストマネージャーが主導しますが、**PMと開発リードの合意が不可欠**です。「何を守るか」はビジネス判断であり、「どのレベルで担保するか」は技術判断だからです。テスト戦略は、QAだけのドキュメントではなく、**チームのドキュメント**です。

### Q. 既存のテスト計画書をどう分離すればいい？

既存のテスト計画書の各項目を、「Why / What」と「How / When / Who」に色分けしてみてください。

| 戦略に移動する項目 | 計画に残す項目 |
|---|---|
| テスト目的・品質目標 | スケジュール |
| リスク分析 | 担当者割り当て |
| テストレベルの方針 | テスト環境 |
| テスト終了条件 | テストケース・データ |
| テスト対象外の定義 | 今回固有のリスク対策 |

この色分けを1回やるだけで、次のリリースから「テスト計画は段取りだけ書けばいい」状態になります。

## アジャイルでの落とし穴——マンネリ化を防ぐ

テスト戦略と計画を分けた。戦略は長寿命で、計画はスプリントごと。ここまでは正しい。しかし、アジャイルで実際に運用すると、**別の問題**が出てきます。

> 戦略が「正しく放置」されて、誰も見ない聖典になる。

戦略は「変わりにくい」と説明しました。しかし、「変わりにくい」と「変えなくていい」は違います。プロダクトは進化し、リスクの所在は変わり、チームの構成も変わる。なのに戦略だけが半年前のまま——これが**マンネリ化**の正体です。

### マンネリ化の3つの症状

| 症状 | 現場で聞こえてくる声 | 本質的な問題 |
|---|---|---|
| **戦略の形骸化** | 「あの戦略ドキュメント？まだあるけど、誰も見てないよ」 | 戦略がプロダクトの現在と乖離している |
| **計画のコピペ** | 「前スプリントのテスト計画、コピーして日付変えた」 | リスク認識が更新されていない |
| **テスト範囲の固定化** | 「いつもと同じところをテストしてる。でも安心はできない」 | 変更に応じたテストの濃淡がない |

### 対策1：スプリントごとに「品質リスクトップ3」を更新する

戦略書そのものを毎スプリント書き直す必要はありません。代わりに、スプリント計画の中で**「今回の品質リスクトップ3」を毎回チームで確認**します。

![アジャイルでの品質サイクル](/images/test-plan-vs-strategy/agile-quality-cycle.png)

:::details スプリント計画会での5分間
**QA：** 「今スプリントのリスクトップ3です。1つ目、決済APIの外部連携先が変更されます。2つ目、認証フローにリファクタが入ります。3つ目、新しいチームメンバーが実装する画面が2つあります」

**開発：** 「認証のリファクタ、影響範囲が広いから重点テストにしたほうがいいね」

**PM：** 「決済の外部連携、障害時のフォールバックも確認してほしい」
:::

これだけで、戦略の方針は変えずに、**リスク認識だけが毎スプリント更新**されます。戦略が自動的に「再解釈」され、形骸化を防げます。

### 対策2：テストケースではなく「テストチャーター」で動く

テストケース一覧を毎スプリント同じ構成で回していると、テスターの思考が固定化します。代わりに、**テストチャーター**（探索すべきテーマを短文で定義したもの）で動く方法があります。

| | 固定化するテスト | チャーターで動くテスト |
|---|---|---|
| **内容** | 「ログインテスト：ID入力→パスワード入力→ログインボタン→ホーム画面確認」 | 「認証リファクタ後のセッション管理を探索する。特に、複数デバイスでの同時ログイン時の挙動に注目」 |
| **結果** | 毎回同じ手順。新しいリスクは見つからない | 今回のリスクに合わせた切り口。同じ領域でも発見が変わる |

テストチャーターは、戦略のリスク方針とスプリントの変更内容を掛け合わせて作ります。同じ「認証」領域でも、スプリントの変更内容が違えばチャーターも変わる。これが**マンネリ化の自然な解消**につながります。

### 対策3：品質ふりかえりを5分だけやる

スプリントのふりかえり（レトロスペクティブ）に、**品質の観点を5分だけ追加**します。

1. **今スプリントのテストで何がわかったか？**
   バグの傾向、テストの効率、想定外の発見。事実を共有する。

2. **リスク認識を変えるべきか？**
   「この領域は思ったより安定していた」「逆にここは要注意」。次スプリントのリスクトップ3に反映する。

3. **テストの方法を変えるべきか？**
   「この領域は自動化に移行できる」「探索的テストの時間を増やしたい」。計画のやり方を進化させる。

この3つの問いを毎スプリント回すだけで、テスト戦略は**「書いたら終わり」ではなく「チームの議論から常に更新されるもの」**に変わります。

:::message
**戦略は「変えない」のではなく「方針は保ちつつリスク認識を更新する」**
テスト戦略の方針（何を守るか）はそう簡単に変わりません。でも、リスクの所在は毎スプリント変わります。方針は固定、リスクは更新——この使い分けが、アジャイルでのテスト戦略運用の核心です。
:::

### まとめると

| 対策 | 頻度 | 所要時間 | 効果 |
|---|---|---|---|
| **品質リスクトップ3** | 毎スプリント計画時 | 5分 | 戦略の再解釈。リスク認識の自動更新 |
| **テストチャーター** | 毎スプリント | 設計時に組み込み | テストの切り口を変え、思考の固定化を防ぐ |
| **品質ふりかえり** | 毎スプリントレトロ | 5分 | テスト方法の継続的改善。戦略の生存確認 |

どれも大掛かりな仕組みではありません。**既存のスクラムイベントに5分を足すだけ**。それだけで、テスト戦略は「死んだドキュメント」から「生きた方針」に変わります。

## まとめ

テスト戦略とテスト計画の違いは、突き詰めるとシンプルです。

**テスト戦略 = 「何を守るか」の方針**
プロジェクトを通じて一貫する。PMと合意する。変わりにくい。

**テスト計画 = 「どうやるか」の段取り**
リリースやスプリントごとに更新する。テストチームが主導する。変わりやすい。

この2つが混同されていると、テスト計画は毎回ゼロから作り直され、テスト方針はリリースごとにブレ、「何のためにテストしているのか」が誰にもわからなくなります。

逆に、この2つがきちんと分かれていると、テストチームは「方針は決まっている。今回は段取りを考えるだけ」と迷いなく動けます。PMは「方針は合意済み。進捗だけ確認すればいい」と安心できます。

**分けるだけで、チーム全体が楽になる。**今さら聞けないことほど、早めに整理する価値があります。

> 戦略は「なぜ登るか」。計画は「どう登るか」。
> この2つを混ぜたまま山に登ると、頂上にたどり着けない。
