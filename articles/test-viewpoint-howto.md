**テスト観点が書けない——その壁を壊す**

「テスト観点で語れ」「足りない観点を指摘しなさい」——[前回の記事](https://zenn.dev/93ch/articles/trust-building-for-testers)でそう書きました。でも、こう思った人もいるんじゃないでしょうか。

「で、テスト観点ってどうやって書くの？」

テスト観点という言葉は知っている。大事なのもわかる。でも、いざ書こうとすると手が止まる。何を書けばいいのか、どこまで書けばいいのか、これで合ってるのか。その不安が、最初の一歩を阻んでいます。

この記事では、テスト観点が書けない原因を3つの壁に分解して、それぞれの乗り越え方を具体例付きで紹介します。

この記事の手順を踏むと、最終的にこういうアウトプットが作れるようになります。

| 大観点 | 中観点 | 小観点 | テスト観点 | 追加理由 | 追加日 |
|---|---|---|---|---|---|
| 入力値の妥当性 | 文字種チェック | 絵文字の受け入れ | 氏名フィールドに絵文字を入力した場合、適切にバリデーションされること | 本番で画面崩れ発生 | 2026-04-15 |
| 入力値の妥当性 | 文字数チェック | 最大文字数・境界値 | 姓を50文字・51文字で入力した場合の挙動 | DBスキーマから導出 | 2026-05-01 |
| 状態遷移 | 異常フロー | ブラウザバック | 確認画面からブラウザバックした場合、入力値が保持されること | 画面遷移図から導出 | 2026-05-01 |
| データ整合性 | 一意性 | メールアドレス重複 | 登録済みメールアドレスで再登録した場合、エラーが表示されること | 仕様書から導出 | 2026-05-01 |
| セキュリティ | インジェクション | XSS | 氏名フィールドにスクリプトタグを入力した場合、無害化されること | AIの壁打ちで追加 | 2026-05-20 |
| パフォーマンス | 同時処理 | 大量同時登録 | 100件同時登録リクエスト時の応答時間とデータ整合性 | チームWSで追加 | 2026-06-01 |

白紙の状態からここに至るまでの道筋を、順を追って説明します。

## テスト観点とは何か

テスト観点とは、「何を、どの視点で、どんな条件でテストするか」を構造化したもの。テストケースの上位概念で、テスト設計の骨格です。

詳しくは[テスト観点を軸にした品質サイクルのすすめ](https://zenn.dev/93ch/articles/test-viewpoint-cycle)にまとめています。ここでは「書き方」に集中します。

## 壁①：何を書けばいいかわからない

一番多い壁です。白紙のドキュメントを前に「テスト観点を書いてください」と言われても、何から手をつければいいかわからない。

この壁には2つの入口があります。

### 入口A：画面から始める

仕様書を読み込む前に、まずテスト対象の画面を開いてみてください。

たとえば、アカウント作成画面。目に見える要素を片っ端から列挙するところから始めます。

| 画面要素 | 何ができる？ | どうなったらまずい？ |
|---|---|---|
| 氏名（姓） | テキスト入力 | 空のまま登録、全角半角混在、スペースだけ、記号入力、極端に長い名前 |
| 氏名（名） | テキスト入力 | 姓と同様。加えて、姓だけ入力して名が空のケース |
| カナ（姓） | テキスト入力 | 半角カナで入力、長音「ー」の扱い、小文字カナ「ァィゥ」 |
| カナ（名） | テキスト入力 | カナ以外の文字（漢字、英字）が入力されたら？ |
| 生年月日 | 日付選択 or 入力 | 未来の日付、閏年2月29日、存在しない日付（4/31）、未成年の年齢制限 |
| 登録ボタン | 押下 | 連打、必須項目が空の状態で押下、通信中に再押下 |

これだけで、テスト観点の種がたくさん出てきます。「氏名」ひとつとっても、文字種・文字数・空入力・境界値と、複数の観点が含まれている。

画面要素 → できること → まずいこと。この3ステップで観点は自然に生まれます。仕様書がなくても、薄くても、画面さえあれば始められる。

### 入口B：仕様書から始める

仕様書がある場合は、名詞と動詞を抜き出す方法が有効です。

仕様書の一文を取り出します。

> 「ユーザーはメールアドレスとパスワードを入力してアカウントを作成する」

| 名詞（何を） | 動詞（どうする） | 観点の種 |
|---|---|---|
| ユーザー | — | 新規ユーザー、既存ユーザー、退会済みユーザー |
| メールアドレス | 入力する | 形式不正、既に登録済み、大文字小文字、ドメインの種類 |
| パスワード | 入力する | 最小文字数、最大文字数、使用可能な記号、コピペ可否 |
| アカウント | 作成する | 作成成功、作成失敗、途中離脱、二重作成 |

仕様書の文章をこうやって分解すると、「テスト対象」と「テスト観点」が同時に見えてきます。

### どちらから始めるべきか

どちらでもいい。大事なのは手を動かすことです。

仕様書が整備されているなら入口Bが効率的。仕様書がない、または薄い環境なら入口Aで画面から始める。両方やれば、片方だけでは見落とす観点が拾えます。

### Before / After：手法を使うと何が変わるか

| Before（漠然とした観点） | After（この手法で言語化した観点） |
|---|---|
| 「アカウント作成まわりをテストする」 | 「氏名フィールドの文字種チェック：全角・半角・記号・絵文字の受け入れ」 |
| 「入力チェックを確認する」 | 「パスワードの文字数チェック：最小8文字・最大72文字・境界値」 |
| 「異常系もやる」 | 「登録ボタンの多重送信：通信中に再押下した場合の二重作成防止」 |

Beforeの状態では、テストケースに落とせない。Afterまで具体化すると、何をテストすればいいかが一意に決まる。この差を生むのが、画面要素の列挙と仕様書の名詞・動詞分解です。

## 壁②：粒度が決められない

観点は出せるようになった。でも今度は「これ、細かすぎない？」「逆に大きすぎない？」で悩む。

### 大・中・小の3層で考える

観点の粒度は、大・中・小の3層に整理すると迷わなくなります。

アカウント作成画面を例にすると：

| 大観点 | 中観点 | 小観点（例） |
|---|---|---|
| 入力値の妥当性 | 文字種チェック | 全角・半角・記号・絵文字の受け入れ |
| 入力値の妥当性 | 文字数チェック | 最小文字数・最大文字数・境界値 |
| 入力値の妥当性 | 必須チェック | 空入力・スペースのみ・未入力での登録 |
| 状態遷移 | 正常フロー | 入力→確認→完了の一連の遷移 |
| 状態遷移 | 異常フロー | 途中離脱、戻るボタン、ブラウザバック |
| データ整合性 | 一意性 | メールアドレスの重複チェック |
| データ整合性 | 保存の正確性 | 入力した値がDBに正しく保存されるか |

正解の粒度はありません。チームで「この粒度で通じるか」を確認しながら調整していくものです。

### 粒度がバラバラになったら

一人で書いていると粒度がバラつきます。これは避けられません。だから、チームでやるのが効果的です（後述）。

## 壁③：網羅性に自信がない

「これで全部？抜けてない？」——この不安が、テスト観点を書くことへの最大の心理的ブレーキです。

結論から言います。完璧な網羅は不可能です。そして、完璧である必要もありません。

大事なのは「出せるだけ出して、あとから育てる」こと。最初から完璧を目指すと手が止まります。ここでは網羅性を高める3つの方法を紹介します。

### 既存ドキュメントをAIに読み込ませて観点を生成する

壁②で粒度の感覚を掴んだら、ゼロから自分で書き出す必要はありません。チームには既にドキュメントがあるはずです。それをAIに読み込ませて、観点の叩き台を作ってもらう方法が有効です。

使えるドキュメントの例：

| ドキュメント | AIが拾ってくれる観点 |
|---|---|
| マニュアル・ヘルプ | ユーザーが実際に触る操作フロー、正常系の網羅 |
| DBスキーマ | 項目のNOT NULL制約、文字数上限、型の境界値、テーブル間の整合性 |
| Figma（画面デザイン） | 画面要素の一覧、入力フィールドの種類、バリデーション表示 |
| Figma（画面遷移図） | 遷移パターン、分岐条件、戻る・中断時の挙動 |

プロンプト例：

```
以下はアカウント作成機能のDBスキーマです。
このスキーマから、テスト観点を大・中・小の3層で抽出してください。

【スキーマ】
users テーブル
- last_name: VARCHAR(50) NOT NULL
- first_name: VARCHAR(50) NOT NULL
- last_name_kana: VARCHAR(100) NOT NULL
- first_name_kana: VARCHAR(100) NOT NULL
- birth_date: DATE NOT NULL
- email: VARCHAR(255) NOT NULL UNIQUE
- ...
```

DBスキーマを渡すと、「VARCHAR(50)だから50文字の境界値テスト」「NOT NULLだから空入力のバリデーション確認」「UNIQUEだから重複登録」といった観点が、制約に基づいて出てきます。自分で仕様書を読み込んで洗い出すより速いし、見落としにくい。

Figmaの画面遷移図を渡せば、「確認画面から戻ったときの入力値保持」「途中離脱→再開時の挙動」など、画面単体では見えにくい遷移系の観点も拾えます。

### AIに壁打ちして抜け漏れをチェックする

叩き台ができたら、自分やチームで出した観点と突き合わせて、抜け漏れをチェックします。

プロンプト例：

```
以下はアカウント作成画面のテスト観点です。
抜けている観点があれば指摘してください。

【対象】アカウント作成画面（氏名、カナ、生年月日、メールアドレス、パスワード）

【現在の観点】
- 入力値の妥当性（文字種、文字数、必須チェック）
- 状態遷移（正常フロー、異常フロー）
- データ整合性（一意性、保存の正確性）
```

AIは「セキュリティ（SQLインジェクション、XSS）」「アクセシビリティ」「国際化（多言語対応、タイムゾーン）」「パフォーマンス（大量同時登録）」といった、自分だけでは思いつきにくい切り口を返してくれます。

AIの出力をそのまま使う必要はありません。「あ、この視点は抜けてた」と気づければ十分。生成と壁打ちの2段階で使うのがちょうどいい距離感です。

### チームで出し合う

3つ目の方法は、チームで観点を出し合うことです。

一人の視点には限界があります。開発者は実装の落とし穴を知っている。企画はユーザーの使い方を知っている。CSはよくある問い合わせを知っている。それぞれの視点から出た観点を集めると、一人では到達できない網羅性が生まれます。

## 一人で抱えない——チームで観点を育てる

テスト観点は、一人で完成させるものではありません。チームで出し合い、磨き合うものです。

### 15分でできる「観点出しワークショップ」

大掛かりな会議は不要です。15分あれば十分。

**進め方：**

1. **お題を決める**（2分）
   次にテストする機能や画面をひとつ選ぶ。

2. **各自で観点を書き出す**（5分）
   付箋でもスプレッドシートでもいい。一人3〜5個。考え込まずに「思いつくまま」で。

3. **共有して整理する**（5分）
   似たものをまとめ、大・中・小の粒度を揃える。

4. **抜けを議論する**（3分）
   「他にない？」と一巡。ここでAIの出力を見せるのも効果的。

このワークを2〜3回やると、チームの中に「観点を出す筋力」がついてきます。

### 出した観点をカタログにする

ワークショップで出した観点は、そのまま放置すると消えます。カタログとして蓄積する仕組みが必要です。

形式はシンプルでいい。スプレッドシートでもNotionでも、チームが見られる場所にあれば十分。大事なのは「ここを見れば観点がある」という場所を作ること。

| 大観点 | 中観点 | 小観点 | 追加理由 | 追加日 |
|---|---|---|---|---|
| 入力値の妥当性 | 文字種チェック | 絵文字の受け入れ | 本番で絵文字入力による画面崩れが発生 | 2026-06-01 |

「追加理由」が地味に大事です。なぜその観点が必要なのかが残っていれば、後から見た人も納得できる。バグがきっかけで追加された観点なら、その不具合情報をリンクしておくとさらに良い。

## カタログは「作って終わり」じゃない

テスト観点カタログの最大の落とし穴は、作った直後がピークで、あとは誰も見なくなること。

カタログが生き続けるには、更新される仕組みが必要です。

### バグが出たら観点を追加する

本番でバグが出た。そのバグは、既存のどの観点で防げたはずだったか？ どの観点があれば見つけられたか？

この振り返りをバグ修正とセットでやるだけで、カタログは自然に育ちます。「バグ→観点追加」を習慣にすること。

### テストが空振りしたら観点を見直す

何十件もテストして、一件もバグが出ない観点がある。それは品質が高いのか、観点がズレているのか。

空振りが続く観点は、粒度を見直すか、優先度を下げるか、別の観点に吸収するか。カタログを「減らす」メンテナンスも大事です。

### 更新のタイミングを決めておく

「気づいたら更新する」は機能しません。タイミングを決めておくのが現実的です。

- スプリントの振り返りで5分だけ「観点の追加・修正」の時間を取る
- バグ修正のチケットに「観点カタログの更新」をチェック項目として入れる
- 月に1回、カタログを眺めて「これまだ要る？」と棚卸しする

仕組みにしてしまえば、属人化しない。誰かがサボっても、仕組みが回る。

## まず3つ、書いてみる

テスト観点を書けるようになるのに、特別なスキルは要りません。

画面を開いて要素を列挙する。仕様書から名詞と動詞を抜き出す。「どうなったらまずい？」を考える。迷ったらAIに壁打ちする。チームで出し合う。カタログに貯めて、使いながら育てる。

最初から完璧な観点リストは作れません。でも、3つなら書ける。3つ書けたら、次は5つ。5つ書けたら、チームに見せて「足りない？」と聞く。そこから始まります。
