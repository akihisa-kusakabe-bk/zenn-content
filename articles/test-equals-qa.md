**「テスト＝QA」を卒業する日**

> 「テストしてるからQAやってます」——これ、昔の自分です。

開発エンジニアから第三者検証会社を経て、今はビットキーで品質戦略を担当している。キャリアのどこかで気づいた。**テストとQAは、同じものじゃない。**

この記事は、2026年5月に開催された勉強会「[AI時代に、品質エンジニアとして何者になるか](https://bitkey.connpass.com/event/391877/)」での登壇内容をベースに、**テスト・QC・QAの違い**を現場の言葉で解きほぐしたものだ。

読んでほしいのは、テスト実務者と、QAに興味がある開発者。「テストとQAって違うの？」と思った人にこそ、届けたい。

## テスト＝QAだと思っていた頃

正直に言う。自分もずっと「テスト＝QA」だと思っていた。

テストケースを書いて、実行して、バグを見つけて、報告する。これが品質を守る仕事だと。テスト項目を増やせば品質が上がる。カバレッジを100%にすれば安心。そう信じていた時期がある。

でも、あるとき気づく。

- テストを増やしても、同じようなバグが出続ける
- リリースのたびに「前回と同じパターン」の障害が起きる
- テストケース数は立派なのに、ユーザーからの問い合わせは減らない

**テストは「やっている」のに、品質は「上がっていない」。** この矛盾に気づいたとき、テスト＝QAという等式が崩れ始めた。

## 「勉強のたとえ」で考える

テスト・QC・QAの違いを、登壇では**勉強のたとえ**で説明した。これが一番刺さったらしいので、深掘りする。

### テスト＝問題を解くこと

数学のテストを思い出してほしい。問題用紙が配られて、制限時間内に解く。テストが終わったら答案を提出する。

ソフトウェアのテストも同じだ。

| 勉強 | ソフトウェア |
|---|---|
| 問題を解く | テストケースを実行する |
| 正解か不正解かわかる | Pass / Fail がわかる |
| 点数が出る | バグの件数が出る |

テストは**「今の状態を測定する行為」**だ。問題を解いただけでは、頭は良くならない。テストを実行しただけでは、品質は上がらない。

### QC（品質管理）＝答え合わせして間違いを直すこと

テストが終わったら、答え合わせをする。間違えた問題を見直して、なぜ間違えたかを考えて、正しい解き方を覚え直す。

これがQC——品質管理だ。

| 勉強 | ソフトウェア |
|---|---|
| 答え合わせする | バグを分析する |
| 間違いの原因を考える | 根本原因を特定する |
| 正しい解き方を覚え直す | バグを修正する |
| 似た問題を解き直す | 修正後にリグレッションテストする |

QCは**「見つけた問題を潰す行為」**だ。バグを見つけて、直して、確認する。多くの現場で「QA」と呼ばれている活動は、実はこのQCであることが多い。

ここまでは、**過去と現在**を見ている。

### QA（品質保証）＝勉強の仕方そのものを改善すること

ここからが本題。

成績を本気で上げたい人は何をするか。問題を解くことでも、答え合わせすることでもない。**勉強の仕方そのもの**を見直す。

- 朝と夜、どっちの方が集中できる？
- この教科は教科書を読むより、問題集から入った方が効率いいのでは？
- そもそも、試験範囲の重点ポイントを押さえてから勉強すべきでは？
- ノートの取り方を変えたら、復習の効率が上がるんじゃないか？

これがQA——品質保証だ。

| 勉強 | ソフトウェア |
|---|---|
| 勉強法を見直す | 開発プロセスを改善する |
| 効率の良い順番を考える | テスト戦略を設計する |
| 苦手分野を重点的にやる | リスクベースでテストを配分する |
| 先生に質問しやすい環境をつくる | 開発者がバグを作り込みにくい仕組みをつくる |

QAは**「良いものが生まれる仕組みをつくる行為」**だ。未来を見ている。

## 3つの違いを一枚で

```
テスト  →  測る     →  「今どうなっているか」を知る
QC     →  直す     →  「見つけた問題」を潰す
QA     →  仕組む   →  「問題が起きにくい状態」をつくる
```

テストはQAの一部だ。QCもQAの一部だ。でも、**テスト＝QAではない。**

テストだけやって「QAやってます」と言うのは、問題を解いただけで「勉強してます」と言うようなものだ。答え合わせも、勉強法の改善もしていないのに。

## 現場で見る「テスト＝QA」の症状

「テスト＝QA」のままだとどうなるか。よくある症状を挙げる。

**症状①：テストケースは増えるが品質は変わらない**
毎回テスト項目を追加するが、バグの傾向は変わらない。問題を解く量は増えているが、勉強法を変えていないので成績は横ばい。

**症状②：同じパターンのバグが繰り返される**
「またこの手のバグか」が口癖になる。答え合わせまではしているが、根本的な原因（＝勉強法の問題）に手をつけていない。

**症状③：リリース直前にしか品質の話をしない**
テスト＝QAだと、テストフェーズでしか品質に関われない。設計段階やコードレビューの段階で品質に関与する発想がない。試験前日にしか勉強しないのと同じ。

**症状④：「QAチーム」がテスト実行チームになっている**
QAという名前がついているのに、やっていることはテスト実行と不具合報告だけ。戦略もプロセス改善もない。名前と実態が乖離している。

## 卒業するために

「テスト＝QA」を卒業するのに、大きな改革は要らない。

### まず、言葉を使い分ける

「テストする」と「QAする」を意識的に使い分けるだけで、思考が変わる。「今やっているのはテストなのか、QCなのか、QAなのか」と自分に問いかける。

### テストの「後」に目を向ける

テスト結果をバグ報告で終わらせない。「なぜこのバグが生まれたのか」「どうすれば防げたのか」まで考える。答え合わせの先に進む。

### 「上流」に関わる

要件定義や設計の段階で「テストしやすさ」や「リスクの高い箇所」について意見する。テストフェーズを待たずに品質に関与する。試験範囲を確認してから勉強を始めるように。

### 仕組みに投資する

一回のテストを頑張るより、テストの仕組みを良くすることに時間を使う。自動テスト、CI/CD、テスト観点の整理、ナレッジの蓄積。勉強法の改善に投資する。

## おわりに

「テスト＝QA」だと思っていた過去の自分に言いたい。

**テストはQAの一部分。大事な一部分だけど、全部じゃない。**

問題を解くことは勉強の入り口であって、ゴールじゃない。答え合わせをして、勉強の仕方を見直して、初めて「勉強している」と言える。

テストをして、結果を分析して、プロセスを改善して、初めて「QAしている」と言える。

この違いに気づいた日が、「テスト＝QA」を卒業する日だ。
