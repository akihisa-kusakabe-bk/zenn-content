**初めてテスト戦略に関わるあなたへ——迷わない地図を渡します**

「テスト戦略を作ってほしい」——ある日、そう言われたらどうしますか？

テスト計画なら書いたことがある。テストケースなら作れる。でも「戦略」と言われると、途端に手が止まる。何を、どこまで、どういう順番で考えればいいのかわからない。

安心してください。テスト戦略は、天才的なひらめきで作るものではありません。**正しい問いを、正しい順番で考えていくフレームワーク**です。この記事では、そのフレームワークを1から、省略なしで解説します。

:::message
**この記事の対象者**
- 初めてテスト戦略の策定に関わるQAエンジニア
- テストチームと一緒にプロジェクトを進めるPM
- 「テスト計画は書けるが、戦略との違いがわからない」人
:::

## テスト戦略とテスト計画は違う

最初に、最も多い混同を解消します。テスト戦略とテスト計画は**別のもの**です。

| | テスト戦略 | テスト計画 |
|---|---|---|
| 決めること | 「何を守るか」 | 「どうやるか」 |
| 視点 | リスクに基づく優先順位づけ | スケジュール・担当・環境の設定 |
| スコープ | プロジェクト全体を見渡す | 個別のテスト活動を計画する |
| 定義するもの | Why と What | How と When |
| 例 | 「決済の信頼性を最優先で担保する」 | 「第3週に決済APIの結合テストを実施する」 |

**戦略なきテスト計画は、地図なしのドライブ**です。目的地（何を守るか）を決めずにルート（どうやるか）だけ決めても、正しい場所にはたどり着けません。

> テスト計画を書く前に、テスト戦略がある。
> この順番を間違えると、すべてがズレる。

## テスト戦略の全体像——5つのステップ

テスト戦略は、以下の5つのステップで構成されます。上から順番に考えていくことで、抜け漏れなく戦略を組み立てられます。

![テスト戦略の5ステップ](/images/test-strategy-101/strategy-5steps.png)

それぞれのステップを、具体例を交えて解説していきます。

## Step 1：テストの目的を定義する

最初のステップは、意外かもしれませんが「そもそもなぜテストするのか」を明文化することです。

### 「品質を上げるため」は目的ではない

「品質を上げるためにテストする」——これは目的のように見えて、実は何も言っていません。**品質のどの側面を、どの水準まで上げるのか**を具体化して、初めて目的になります。

:::message alert
**曖昧な目的：「品質を担保する」「バグをなくす」**
抽象的すぎて、テスト範囲もゴールも決められない。結果として「全部テストしないと不安」に陥る。
:::

:::message
**具体的な目的：「決済処理の信頼性を担保し、リリース後の決済エラー率を0.1%未満に維持する」**
対象（決済処理）、品質特性（信頼性）、基準（0.1%未満）が明確。テストの範囲と深さが自然に決まる。
:::

### 目的を定義するための3つの問い

1. **このプロダクトで「壊れたら最も困る」機能はどれか？**
   ビジネス上のインパクトが最大の機能を特定する。PMや事業責任者に聞くのが最も確実。

2. **ユーザーが「これだけは動いてほしい」と思う体験は何か？**
   機能単位ではなく、ユーザー体験の観点で考える。「ログインできる」「購入完了まで到達できる」など。

3. **過去のリリースで問題になったことは何か？**
   過去のインシデントや障害報告を見る。同じ失敗を繰り返さないことは、戦略の最低ライン。

:::message
**PMへのアドバイス**
テストの目的を決めるのは、テスターだけの仕事ではありません。「何を守りたいか」はビジネス判断です。PMがこの問いに答えることで、テスト戦略はプロダクト戦略と整合します。
:::

## Step 2：リスクを特定する

目的が定まったら、次は**「何がその目的を脅かすか」**を特定します。これがリスク分析です。

### リスクの2つの軸

リスクは「影響度」と「発生確率」の2軸で評価します。両方が高いものが、テストで最優先に扱うべき領域です。

| | 影響度：大 | 影響度：小 |
|---|---|---|
| **発生確率：高** | **🔴 最優先** — 深く、広くテストする。探索的テストも併用 | **標準** — 主要パスを確認 |
| **発生確率：低** | **重点確認** — 障害時シナリオを中心に | **🟢 軽量** — スモークテスト or 自動化 |

### リスクを見つける4つの情報源

「リスクがどこにあるかわからない」という人のために、確実にリスクが見つかる4つの情報源を紹介します。

**情報源 1：変更差分（PR / コミットログ）**
変更が大きい箇所はリスクが高い。特に、複数の機能にまたがる変更は要注意。

**情報源 2：過去のバグ履歴**
過去にバグが集中した領域は、再びバグが出やすい。バグは偏在する。

**情報源 3：開発者の「ここ怖い」**
開発者が不安に感じている箇所を直接聞く。自信がない部分は、テストで補完すべき部分。

**情報源 4：ユーザーからのフィードバック**
カスタマーサポートへの問い合わせ、アプリレビュー。ユーザーが困っている箇所には、潜在的なリスクがある。

:::message
**「リスクがわからない」のもリスク**
リスク分析に時間をかけすぎる必要はありません。完璧なリスク一覧を作ることが目的ではなく、「最もテストすべき場所」の優先順位を決めることが目的です。30分でリスクトップ5を出す——まずはそこからで十分です。
:::

## Step 3：テストレベルを設計する

リスクが特定できたら、次は**「どのレベルでテストするか」**を決めます。

### テストレベルとは

テストレベルとは、テストを行う粒度のことです。大きく4つに分かれます。

![テストレベルのピラミッド](/images/test-strategy-101/test-levels.png)

### 戦略的に考える：どのレベルでどのリスクを潰すか

テスト戦略で重要なのは、**各レベルの役割を明確にし、重複を避ける**ことです。

| リスク | 適切なテストレベル | 理由 |
|---|---|---|
| 計算ロジックの誤り | **単体テスト** | 境界値・異常値を大量に高速で検証できる |
| API間のデータ不整合 | **結合テスト** | モジュール間のインターフェースで発生する問題を検出 |
| ユーザー操作フローの破綻 | **システムテスト** | 実際のユーザー操作を再現して確認する必要がある |
| ビジネス要件との乖離 | **受け入れテスト** | 技術的に正しくても、要件を満たさなければ意味がない |

:::message alert
**よくある失敗：全部をシステムテストでやろうとする**
すべてのリスクをシステムテストで潰そうとすると、テストが膨大になり、実行時間もコストも爆発します。計算ロジックは単体テスト、連携はAPIテスト——適材適所で分担することが、戦略の核心です。
:::

## Step 4：テスト観点を整理する

テストレベルが決まったら、次は**「何を確認するか」**——つまりテスト観点を整理します。テスト戦略の中で、ここが最も実務に直結するステップです。

### テスト観点とは

テスト観点とは、「テストで確認すべき視点」のことです。テストケースが「具体的な手順」だとすれば、テスト観点は「なぜその手順が必要か」の根拠にあたります。

| テスト観点（Why / What） | テストケース（How） |
|---|---|
| 「同時アクセス時のデータ整合性」 | 「ユーザーAとBが同時に同じ商品を購入する」 |
| → 確認すべき品質特性・リスクの視点 | → 具体的な操作手順と期待結果 |

### 観点を漏らさないためのカテゴリ

テスト観点を網羅的に洗い出すための基本カテゴリを紹介します。すべてのプロジェクトで使える汎用フレームワークです。

| カテゴリ | 観点の例 | 見落としやすい場面 |
|---|---|---|
| **正常系** | 主要機能が仕様通り動作するか | 見落としにくい（ほぼ全員がやる） |
| **異常系** | 不正入力、通信エラー、タイムアウト | 正常系に時間を取られて後回しになりがち |
| **境界値** | 上限・下限、0件、最大件数 | 「普通の値」でテストして満足してしまう |
| **状態遷移** | ステータス変更、画面遷移の順序 | 正常なフローだけ確認して、イレギュラーな遷移を漏らす |
| **データ整合性** | DB登録値の正確性、関連データの連動 | 画面表示だけ確認して、裏側のデータを見ない |
| **セキュリティ** | 認証・認可、入力バリデーション | 「機能テスト」の範囲外と思い込む |
| **性能** | 応答時間、同時接続、大量データ | 機能的に正しければOKと判断してしまう |
| **ユーザビリティ** | 操作の直感性、エラーメッセージの明瞭さ | 「仕様通り」で片付けてしまう |

:::message
**すべてを均等にテストしない**
上記のカテゴリを全部同じ深さでテストする必要はありません。Step 2で特定したリスクに基づいて、**深くやる観点と軽くやる観点を意図的に分ける**。この濃淡をつけることが「戦略」です。
:::

## Step 5：終了条件を決める

テスト戦略の最後にして最も重要なステップ。それが**「テストをいつ終わりにするか」**を事前に決めることです。

### なぜ終了条件が必要なのか

終了条件がないテストは、永遠に終わりません。「もう少しテストしたほうがいいかも」「念のためもう一回」——この不安に際限がないからです。

終了条件は、**テストを「終わらせる」ためではなく、「終わりを合意する」ため**にあります。チーム全員が「ここまでやれば出荷できる」と合意できる基準を、テスト開始前に決めておくのです。

### 終了条件の具体例

**定量的な基準（数字で測れる条件）**
- 重大バグ（Critical / Major）が0件であること
- テストケース消化率が100%であること
- 残存バグの修正計画がすべて合意されていること

**定性的な基準（判断で決める条件）**
- 主要ユーザーシナリオがすべて正常に動作すること
- リスク最優先領域の探索的テストが完了していること
- ステークホルダーがリリースに同意していること

:::details 終了条件がないとどうなるか
**テスター：** 「テスト完了しました」
**PM：** 「本当に大丈夫？もう少しやったほうがよくない？」
**テスター：** 「……じゃあもう少しやります」（終わりが見えない）
:::

:::details 終了条件があるとこうなる
**テスター：** 「事前に合意した終了条件5項目、すべてクリアしました。レポートはこちらです」
**PM：** 「確認しました。リリース判定会議にかけましょう」
:::

> 「テストが終わらない」の正体は、
> 終了条件が決まっていないことにある。

この5ステップを上から順に埋めていけば、テスト戦略の骨格ができあがります。最初から完璧を目指す必要はありません。**まず骨格を作り、チームでレビューし、プロジェクトの中で磨いていく**——この進め方が最も失敗しにくい方法です。

## よくある失敗パターンと対処法

最後に、テスト戦略の策定で初心者が陥りやすい失敗パターンを3つ紹介します。

| 失敗パターン | なぜ起きるか | 対処法 |
|---|---|---|
| **「全部テストする」戦略** | リスク分析をスキップして、不安から全方位をカバーしようとする | Step 2に戻ってリスクトップ5を決める。「テストしない」判断も戦略 |
| **戦略が「ドキュメント」で終わる** | 立派な戦略書を作ったが、日々のテスト活動と結びついていない | 戦略は1ページに要約する。毎スプリントのテスト計画で「戦略のどの部分を実行しているか」を確認する |
| **テスターだけで戦略を決める** | 開発者やPMを巻き込まず、テストチームだけで完結させてしまう | Step 1（目的）とStep 5（終了条件）は必ずステークホルダーと合意する。テストの目的はビジネスの目的と一致していなければ意味がない |

## まとめ：5つのステップを、まず1回やってみる

テスト戦略は、難しいものではありません。**5つの問いに順番に答えていく**だけです。

1. **何を守るか？** — テストの目的を、品質特性と基準で具体化する
2. **何が脅威か？** — リスクを特定し、テストの優先順位をつける
3. **どのレベルで守るか？** — リスクに応じて、適切なテストレベルに振り分ける
4. **何を確認するか？** — テスト観点を整理し、戦略的に濃淡をつける
5. **いつ終わるか？** — 終了条件を事前に合意し、テストに「終わり」を作る

完璧な戦略を最初から作ろうとしないでください。**まず1回やってみて、チームでレビューして、次に活かす**。この繰り返しが、あなたのチームのテスト戦略を育てていきます。

> テスト戦略とは、「全部やる」の対極にある。
> 「何を守り、何を捨てるか」を決める覚悟のことだ。
